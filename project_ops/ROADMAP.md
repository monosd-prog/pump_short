## Vision

Система `pump_short` — это единый контур для:
- детектирования пампов и формирования сигналов входа в шорт,
- безопасного исполнения сделок (paper и live),
- системного трекинга исходов и рисков,
- построения качественного датасета для последующего ML-фильтра входа.

Цель — получить **устойчиво прибыльную, прозрачно валидируемую** торговую систему, в которой:
- торговля (live или paper) практически всегда активна, чтобы накапливать репрезентативный датасет для факторного анализа,
- каждый сигнал и сделка проходят через явный risk guard,
- каждый исход фиксируется и анализируется,
- качество датасета и метрик контролируется автоматически,
- любые изменения проходят через воспроизводимую валидацию,
- поверх всего этого можно накладывать адаптивные режимы (подрежимы, AI-фильтры, adaptive risk scaling) без потери управляемости.

## Current state

- MVP short-pump стратегии запущен и стабильно крутится на VPS 24/7.
- Есть рабочие контуры:
  - прием сигналов `/pump`,
  - watcher / runtime для short_pump и long_pullback,
  - ядро торговли (`trading.runner`, paper/live, queue, risk),
  - outcome tracking и запись `events_v3/trades_v3/outcomes_v3`,
  - Telegram-уведомления и отчеты,
  - аналитика (`analytics/*`, factor report, executive report).
- Основной фокус сейчас — **сбор датасета** и **первичный анализ статистики**.
- ML-фильтр входа еще не внедрен; входы определяются правилами и простыми фильтрами (stage, dist_to_peak, liq, etc).

## Near-term goals (2–4 недели)

- **Signal Quality**
  - Уточнить контракт сигнала (обязательные/опциональные поля, валидация).
  - Добавить базовый smoke-набор для сигналов (валидность символа, таймстемпа, pump_pct).
  - Ввести метрики качества сигналов (accepted/rejected, причины отказа).

- **Risk Guard**
  - Зафиксировать и документировать текущие правила риск-контроля (size, leverage, max concurrent, cooldown).
  - Добавить автоматизированные проверки для ключевых risk-ограничений (unit-тесты + smoke).
  - Включить базовую авто-проверку ENV/конфига перед деплоем (ограничения плеча, размера позиции).

- **Outcome Tracking / Dataset Quality**
  - Проверить целостность пайплайна `events_v3/trades_v3/outcomes_v3` для актуальных дат.
  - Внедрить простые data-quality проверки (нет дубликатов, нет NaN в критичных полях, корректные связи event↔trade↔outcome).
  - Ввести минимум 2–3 стандартных validation-сценария на выборочной дате (ручная выборка + автосмоки).
  - Проверить, какие индикаторы не пишутся в `events_v3`, и скорректировать сбор так, чтобы lab/paper/live режимы имели сопоставимые поля.

- **Validation / Backtest**
  - Описать стандартный сценарий ручной валидации для новой гипотезы (набор команд, чеклист).
  - Подготовить минимальный offline-набор данных для быстрой локальной проверки (subset датасета).
  - Сформировать первый вариант дневного factor-отчета (на основе текущей "тетради") с основными срезами и дельтами относительно предыдущего значения и 5m-окна.

## Mid-term goals (1–2 месяца)

- **Signal Quality / ML Readiness**
  - Формализовать фичи и таргеты для ML-модели отбора входов (feature contract, схемы).
  - Обеспечить стабильность и непротиворечивость датасета по версиям (dataset schema v3+).
  - Внедрить базовый ML-бейзлайн (например, логистическая регрессия / простой бустинг) поверх текущего фильтра.
  - Определить критерии объема выборки (например, ≥150 сделок на подрежим), при которых имеет смысл строить ML для конкретных режимов.

- **Risk Guard / Production Safety**
  - Разделить конфиги и защиту для paper/live-режимов с явным safety-стратумом (ограничения плеча, максимальный риск в сутки).
  - Настроить алерты по ключевым риск-метрикам (аномальная просадка, всплеск количества сделок, unusual PnL distribution).
  - После стабилизации risk guard (несколько недель работы) спроектировать и описать adaptive risk scaling (например, ACTIVE/LOW HEAT/WATCH/DISABLED режимы с разными коэффициентами риска).

- **Outcome Tracking / Monitoring**
  - Добавить автоматические сравнения paper vs live по одинаковым сетапам (при наличии данных).
  - Ввести мониторинг задержек и пропусков в outcome-трекере (нет исхода дольше заданного SLA).
  - Добавить в отчеты и мониторинг оценку рыночного режима (например, на основе агрегированного funding rate по рынку).

- **Validation / Backtest / Review process**
  - Формализовать стандартный pipeline проверки новой гипотезы: от идеи → до отчета с графиками и метриками.
  - Привязать acceptance criteria к конкретным метрикам (WR, EV, max drawdown, sample size, stability across buckets).
  - Описать крупноблочный процесс внедрения новых стратегий (исторический прогон → paper → включение в live) и связать его с workflow агентов и промптами для AI-инструментов.

---

Вне указанных горизонтов roadmap подразумевает:
- постепенный переход от чисто CSV-хранилища к более структурированной БД с BI/дашбордами (Notion/Superset или аналог),
- развитие "лаборатории" для анализа исторических данных (включая внешние источники типа Coinglass по фандингу/ликвидациям/ОИ),
- внедрение AI-агентов для анализа сделок, поиска закономерностей и предложения новых фильтров/режимов.

## Risks

- **Data Quality Risk**
  - Неполные или конфликтующие datasets (дубликаты, несовпадение событий и исходов).
  - Смена формата CSV или схемы без обновления аналитических/ML-частей.

- **Model / Overfitting Risk**
  - Переобучение ML-фильтра на ограниченном датасете без out-of-sample проверки.
  - Ложное ощущение устойчивости из-за несогласованности train/validation/test-выборок.

- **Operational / Production Risk**
  - Некорректные ENV/конфиги (плечо, размер, режим live) приводят к избыточному риску.
  - Ошибки в логике risk guard или outcome tracking, не покрытые тестами, остаются незамеченными.
  - Telegram/monitoring не отражают реальные сбои (нет алертов при падении сервисов или остановке outcome-трекинга).

- **Process / Governance Risk**
  - Изменения выкатываются без четких acceptance criteria.
  - Отсутствие централизованной фиксации решений (ADR-lite) приводит к расхождению ожиданий.

## Dependencies

- **Биржа и рыночные данные**
  - Доступность API Bybit (klines, ликвидации, позиции, закрытия).
  - Качество и стабильность исторических данных, особенно для редких пампов.

- **Инфраструктура**
  - VPS / сервер с системнымd-конфигами для `pump_short` и `pump_short_analysis`.
  - Хранилище для datasets и логов с достаточным объемом и резервным копированием.

- **Сторонние сервисы**
  - Telegram API для нотификаций и отчетов.
  - Потенциальные внешние сигнальные источники (вебхуки, другие боты).

- **ML / Аналитический стек**
  - Инструменты для анализа (pandas, matplotlib/plotly, и т.д.) и потенциальные ML-библиотеки.
  - Согласованный формат выгрузки датасета для ML-экспериментов.
