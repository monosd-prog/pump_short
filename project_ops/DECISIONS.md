## ADR-lite: Decisions Log

Формат записи:
- **Date** — дата решения (YYYY-MM-DD).
- **Decision** — короткое формулирование.
- **Context** — почему это решение принято, какие были альтернативы.
- **Consequences** — ключевые последствия и trade-offs.

### 2026-03-17 — Единый AI-delivery контур в `project_ops/`

- **Decision**  
  Создать директорию `project_ops/` в корне репозитория `pump_short` и использовать ее как центральное место для операционной документации: roadmap, backlog, роли агентов, workflow, DoD и журнал решений.

- **Context**  
  Проект уже содержит достаточно развитую архитектуру (`PROJECT_STRUCTURE.md`, `docs/ARCHITECTURE_BRIEF.md`, диагностические отчеты), но отсутствует систематизированный контур для управления изменениями, рисками и ML-готовностью. Появляется необходимость в более формальном AI-delivery процессе поверх существующего торгово-аналитического кода.

- **Consequences**  
  - Появляется единая точка входа для операционной документации и принятия решений.  
  - Новые процессы (backlog, workflow, DoD, ADR-lite) становятся явными и могут использоваться как для людей, так и для агентов.  
  - Требуется поддерживать содержимое `project_ops/` в актуальном состоянии при архитектурных изменениях.

### 2026-03-17 — Приоритет data-quality и validation перед ML-фильтром

- **Decision**  
  Перед внедрением продового ML-фильтра входа фокус смещается на повышение качества сигналов, outcome tracking и датасета, а также на формализацию validation-пайплайнов.

- **Context**  
  Текущий статус проекта — рабочий MVP с акцентом на сбор датасета и анализ статистики. Без устойчивого и воспроизводимого пайплайна данных любой ML-слой будет хрупким и склонным к переобучению. Существующие диагностические документы (`docs/DIAG_*.md`) показывают важность корректной обработки edge-cases.

- **Consequences**  
  - В roadmap и backlog приоритет получают задачи по data-quality, outcome tracking и validation.  
  - Включение ML в прод без прохождения минимальных data/validation требований считается нарушением процесса.  
  - Сроки появления стабильного ML-фильтра могут сместиться, но повышается надежность результатов.

### 2026-03-17 — Строгое требование acceptance criteria и validation evidence

- **Decision**  
  Закрепить правило: ни одно изменение в торговой системе не считается завершенным без явных acceptance criteria и зафиксированного validation evidence.

- **Context**  
  Торговая система сочетает высокую сложность (runtime + аналитика + datasets + ML-перспектива) и чувствительность к ошибкам. Без четких критериев готовности и подтвержденных результатов валидации трудно отслеживать регрессии и понимать, действительно ли изменение улучшило систему.

- **Consequences**  
  - Все задачи в `BACKLOG.md` должны содержать измеримые критерии готовности.  
  - QA / Validation Agent получает формальный мандат блокировать релизы без достаточного evidence.  
  - Требуется дополнительное время на подготовку и хранение артефактов проверки, но растет управляемость изменений.

### 2026-03-17 — Разделение ролей агентов для управления риском

- **Decision**  
  Ввести четыре логические роли агентов: Project Manager Agent, Strategy Developer Agent, QA / Validation Agent, Production Safety Agent с четкими зонами ответственности и правилами эскалации.

- **Context**  
  В кодовой базе уже присутствует разделение доменов (runtime, trading, analytics, deploy), но не формализовано, кто за что отвечает на уровне процесса. Для безопасного развития системы (особенно с ML) необходимо развести ответственность за roadmap, реализацию, валидацию и прод-безопасность.

- **Consequences**  
  - Упрощается автоматизация части процессов (каждой задаче назначается Owner Agent).  
  - Становится очевидно, кто может блокировать релиз по каким причинам (QA, Production Safety).  
  - Требуется следить, чтобы изменения, влияющие на риск-политику или ML, всегда проходили через нужные роли.

### 2026-03-17 — Conservative rollout для будущего ML-фильтра

- **Decision**  
  Будущий ML-фильтр входа должен внедряться по консервативной схеме rollout: сначала как аналитический/оффлайн слой, затем как вспомогательный сигнал (gating с фолбеком на rule-based), и только после этого — как основной фильтр при наличии устойчивых результатов.

- **Context**  
  ML-модели для трейдинга подвержены переобучению и зависят от смены рыночного режима. У проекта уже есть работающая rule-based логика, и переход на ML без staged rollout увеличивает production risk.

- **Consequences**  
  - Первые версии ML будут жить как аналитический инструмент (отчеты, сравнения с rule-based).  
  - При переходе в online-гейтинг обязателен фолбек-план и ограничения (например, ML может только уменьшать число сделок, но не увеличивать риск).  
  - Требуется дополнительная инфраструктура для сравнения ML vs rule-based в `analytics/` и мониторинге.
